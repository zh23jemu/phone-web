# 本地从零启动 `phone-web` 项目步骤清单

## 简述
目标是先把项目按“最小可运行”拆成 3 层来启动，再逐步联通：

- 手机端前端：`C:\Coding\phone-web\phone`
- 仿真通话/短信服务：`C:\Coding\phone-web\服务器api.py`
- 账号/后台系统：`C:\Coding\phone-web\likeadmin_php-master`

按当前代码关系，推荐先走两阶段：

- 阶段 1：只启动 `phone`，直接连仓库里写死的远程接口，验证前端能否跑通
- 阶段 2：再本地化 Python 服务、MySQL，最后视需要本地化 likeadmin

## 启动顺序

### 1. 准备基础环境
在 Windows 本机准备这些软件：

- HBuilderX，用来运行 `uni-app` 工程
- Python 3.10+，用于启动 `服务器api.py`
- MySQL 5.7/8.x，用于 `phone` 业务库
- Docker Desktop，可选，用于启动 likeadmin 的 PHP/MySQL/Redis 容器
- 可访问安卓模拟器或安卓真机，用于优先验证 App 端效果

注意点：

- `phone` 不是标准 npm 前端工程，核心运行方式是 HBuilderX
- `phone` 目录下没有常规 `package.json`，不要按 Vite/Node 项目去启动它
- Python 服务代码里数据库连接是写死的本地配置，需保证本地 MySQL 用户和库名一致，或后续改代码配置
- 前端大量接口地址目前写死为 `http://106.53.30.150:2025` 和 `http://106.53.30.150:9097`

### 2. 先做“最小验证”
先不要本地部署后端，直接验证前端是否能跑：

- 用 HBuilderX 打开 `C:\Coding\phone-web\phone`
- 运行到 Android 模拟器或真机
- 先测试登录页是否能打开
- 再测试能否登录、进入拨号页、查看通话记录和短信页

成功标准：

- App 能正常启动
- 登录接口 `:2025` 可访问
- 通话/短信接口 `:9097` 可访问
- 不出现白屏或接口全部报错

如果这一步通过，说明远程依赖还活着，前端已可开发；本地化后端只是在做去远程依赖。

## 前端启动步骤

### 3. 启动 `phone` 前端
工程目录：

- `C:\Coding\phone-web\phone`

操作步骤：

1. 打开 HBuilderX
2. 选择“打开目录”，打开 `C:\Coding\phone-web\phone`
3. 等待 HBuilderX 识别为 `uni-app` 工程
4. 检查入口文件是否正常：
   - `App.vue`
   - `main.js`
   - `pages.json`
   - `manifest.json`
5. 运行到目标设备：
   - 优先：运行到 Android App
   - 备选：运行到浏览器，仅做页面验证

建议优先验证的页面路径：

- 登录页 `pages/auth/login`
- 拨号页 `pages/dial/dial`
- 通话页 `pages/call/calling`
- 短信会话页 `pages/message/message`
- 聊天页 `pages/message/chat`

前端当前依赖的关键接口：

- 登录：`/api/login/account`
- 用户中心：`/api/user/center`
- 拨号：`/api/dial`
- 通话记录：`/api/call-records`
- 来电轮询：`/api/check-incoming-calls`
- 联系人：`/api/contacts`、`/api/contact`
- 会话列表：`/api/conversations`
- 消息列表：`/api/messages`
- 发短信：`/api/send-message`

## Python 服务本地化步骤

### 4. 准备 Python 虚拟环境
在仓库根目录 `C:\Coding\phone-web` 下准备 `.venv`：

1. 创建虚拟环境
2. 用 `.venv\Scripts\python.exe` 安装依赖
3. 后续始终用 `.venv\Scripts\python.exe` 运行脚本

按代码推断需要的核心依赖：

- `websockets`
- `mysql-connector-python`
- `requests`

如果没有依赖文件，按最小集合手动安装即可。

### 5. 准备 MySQL 数据库
Python 服务默认连接信息来自 `服务器api.py`：

- host: `localhost`
- user: `phone`
- password: `phone2025`
- database: `phone`

本地数据库准备步骤：

1. 安装并启动 MySQL
2. 创建数据库 `phone`
3. 创建用户 `phone`
4. 给该用户授予 `phone` 库权限
5. 导入所需表结构

当前从代码能确定至少会用到这些表：

- `la_call_records`
- `la_call_mark`
- `la_contacts`
- `la_messages`
- `la_conversations`

注意：

- 仓库里没有明显现成的 `phone` 数据库初始化 SQL 被直接指向
- 如果没有现成 SQL，需要从 Python 接口代码反推字段建表
- 在正式建表前，应先梳理每个接口依赖的字段，避免频繁补字段

### 6. 启动 Python 服务
主服务建议以 `C:\Coding\phone-web\服务器api.py` 为准。

启动目标：

- WebSocket 服务监听 `9096`
- HTTP API 服务监听 `9097`

启动后应验证：

- `9097` 端口已监听
- 基础 GET 接口有响应
- 前端调用 `call-records`、`contacts`、`messages` 不再报连接失败
- WebSocket 或轮询相关逻辑不报端口不可达

前端本地联调时，需要把 `phone` 内写死的 `106.53.30.150:9097` 改成本机地址，推荐统一抽成配置后再切换，避免逐文件硬改遗漏。

## likeadmin 本地化步骤

### 7. 判断是否必须本地化 likeadmin
当前 `phone` 登录页调用的是：

- `http://106.53.30.150:2025/api/login/account`
- `http://106.53.30.150:2025/api/user/center`

这部分不属于 Python `9097` 服务，因此如果你希望完全离线本地跑，likeadmin 或其等价账号服务也要本地化。

推荐先确认两件事：

- 本地是否只需要“能进系统”，那可以临时绕过登录，直接写入 `authToken` 和 `userId`
- 本地是否必须保留真实登录流程，如果必须，就需要把 `:2025` 的账号服务也搭起来

### 8. 本地启动 likeadmin
likeadmin 根目录：

- `C:\Coding\phone-web\likeadmin_php-master`

推荐方式是用 Docker 先拉起基础环境：

- nginx
- php-fpm
- mysql
- redis

已存在编排文件：

- `C:\Coding\phone-web\likeadmin_php-master\docker\docker-compose.yml`

启动后要完成：

1. 让 likeadmin 的安装页能访问
2. 完成初始化
3. 确认是否提供 `login/account` 与 `user/center` 这类接口
4. 如接口路径或端口与前端不一致，需要做接口映射或前端地址调整

注意：

- likeadmin 自带的 Docker 暴露示例主要是标准后台访问，不保证与你当前 `phone` 中的 `:2025` 完全一致
- 很可能线上 `2025` 端口是另外部署过的 API 网关或改造版服务
- 如果接口不一致，最省事的本地方案通常是先绕过登录，再单独联调 `9097`

## 联调步骤

### 9. 改成本地接口地址
当前前端接口分散写在多个页面里，至少要统一替换两类基地址：

- `106.53.30.150:2025`
- `106.53.30.150:9097`

推荐实现方式：

- 新增统一配置文件，集中维护 `authBaseUrl` 和 `serviceBaseUrl`
- 页面内所有 `uni.request` 改为读配置
- 本地开发使用 `localhost` 或局域网 IP
- 真机调试时优先使用局域网 IP，不要用 `localhost`

### 10. 按功能逐项验证
联调建议按这条顺序验收：

1. 登录页打开正常
2. 登录成功，能写入 `authToken`、`userId`
3. 启动后按登录态跳转正确
4. 拨号页能拉通话记录
5. 输入号码后能进入通话页
6. 通话页能轮询状态、挂断、返回
7. 联系人页能读取联系人
8. 短信会话页能读取 `conversations`
9. 聊天页能读取 `messages`
10. 发送消息能写入并即时显示
11. 来电轮询能命中 `check-incoming-calls`

## 测试与验收

### 11. 最低验收标准
满足以下条件即可认为“本地从零启动”完成：

- `phone` 能在 HBuilderX 中运行
- Python 服务在本地 9097 正常响应
- MySQL 中有可用测试数据
- 拨号记录、联系人、短信会话至少有一项可正常读取
- 能完成一次模拟拨号和一次消息发送
- 如果保留登录链路，本地 `2025` 登录接口也已打通；否则已明确采用绕过登录方案

### 12. 常见阻塞点
最可能卡住的点有这些：

- HBuilderX 可以启动，但接口全部还是连远程地址
- 真机调试使用 `localhost` 导致请求不到本机服务
- Python 服务起来了，但本地 MySQL 表结构不全
- likeadmin 启动成功，但接口路径与 `phone` 代码预期不一致
- `phone` 的部分逻辑依赖真实测试数据，没有种子数据会显示空页面
- WebSocket `9096` 没起，但页面逻辑仍有相关依赖

## 假设与默认选择
本清单基于这些默认假设：

- 以 `C:\Coding\phone-web\phone` 作为主前端工程
- 以 `C:\Coding\phone-web\服务器api.py` 作为主要本地业务服务
- 登录接口与 `9097` 服务分属两个系统
- 优先目标是“先本地跑通主要业务”，不是一开始就 100% 还原线上环境
- 默认推荐先跑前端、再跑 Python 服务、最后再决定是否本地化 likeadmin
- 默认推荐使用 HBuilderX + Android 设备做首轮验证，而不是先折腾浏览器或完整容器化
